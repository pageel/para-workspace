## üåå Overview

**PARA Workspace** is a governed, open-source standard that defines how humans and AI agents organize knowledge and collaborate on projects. It ships as a **repo** containing a kernel (constitution), CLI tools, and templates ‚Äî which generates **workspaces** where you actually work. The kernel enforces 10 invariants and 8 heuristics so every workspace is predictable, auditable, and agent-friendly.

### Three Foundational Principles

1. **Repo ‚â† Workspace** ‚Äî The repo contains governance (kernel, CLI, templates). It never contains user data.
2. **Workspace = Runtime** ‚Äî Generated by `para init`, each workspace is a standalone instance where you and your agent work.
3. **Kernel = Constitution** ‚Äî Immutable rules that all workspaces follow. Changes require an RFC + version bump.

```
Repo     (Constitution + Compiler)
  ‚Üì para init
Workspace (Operating System Runtime)
  ‚Üì agent attach
Agent    (Execution Environment)
```

## üì• Installation

### Quick Start

Open your workspace directory (e.g., in Antigravity or your IDE) and follow these steps:

```bash
# 1. Clone repo into PARA-standard location (from workspace root)
mkdir -p Projects/para-workspace
git clone https://github.com/pageel/para-workspace.git Projects/para-workspace/repo

# 2. Set executable permissions on CLI scripts
chmod +x Projects/para-workspace/repo/cli/para
chmod +x Projects/para-workspace/repo/cli/commands/*.sh

# 3. Initialize your workspace with a profile
./Projects/para-workspace/repo/cli/para init --profile=dev --lang=vi

# 4. Verify everything works
./para status
```

> **What just happened?**
>
> 1. The repo lives at `Projects/para-workspace/repo/` ‚Äî your governance source.
> 2. `chmod +x` ensures all CLI scripts are executable (required on Linux/macOS).
> 3. `para init` creates the PARA directory structure, runs `install.sh` automatically
>    to sync kernel, workflows, governance rules, and generates a `./para` wrapper.
> 4. You can now use `./para` from your workspace root for all commands.

### Updating

```bash
# Pull latest from GitHub and re-sync workspace
./para update
```

This will `git pull` the repo and re-run `install.sh` to sync kernel, workflows, and governance. Existing files are backed up to `.bak` before overwriting.

### What `para init` Does

- ‚úÖ Creates `Projects/`, `Areas/`, `Resources/`, `Archive/` (from profile)
- ‚úÖ Sets **executable permissions** on all CLI scripts
- ‚úÖ Runs **`install.sh`** automatically, which:
  - Installs **kernel snapshot** to `Resources/ai-agents/kernel/`
  - Installs **workflows** to `.agent/workflows/` and catalog
  - Installs **agent governance** rules to `.agent/rules/`
  - Creates **`./para`** wrapper at workspace root
  - Backs up conflicting files to `.bak`
- ‚úÖ Generates **`.para-workspace.yml`** with kernel version tracking

## üß† The Kernel

The Kernel is the **constitution** of PARA Workspace ‚Äî the rules that all workspaces must follow.

### Invariants (Hard Rules ‚Äî change = MAJOR bump)

| #   | Rule                                           |
| --- | ---------------------------------------------- |
| I1  | PARA directory structure is mandatory          |
| I2  | Hybrid 3-file task model (backlog = canonical) |
| I3  | kebab-case project naming                      |
| I4  | No active tasks = inactive project             |
| I5  | Areas contain no runtime tasks                 |
| I6  | Archive is immutable cold storage              |
| I7  | Seeds are raw ideas, not tasks                 |
| I8  | No loose files at workspace root               |
| I9  | Resources are read-only references             |
| I10 | Repo ‚Üî Workspace separation                    |

### Heuristics (Soft Rules ‚Äî change = MINOR/PATCH)

| #   | Guideline                                   |
| --- | ------------------------------------------- |
| H1  | Naming conventions (kebab-case, PascalCase) |
| H2  | Context loading priority                    |
| H3  | Semantic versioning with approval levels    |
| H4  | Standard project directory structure        |
| H5  | Beads lifecycle                             |
| H6  | VCS & Git boundaries                        |
| H7  | Cross-project references via Resources      |
| H8  | Workflow kernel compatibility               |

## üß© Task Management

PARA Workspace uses a **Hybrid 3-File Model**:

```
artifacts/tasks/
‚îú‚îÄ‚îÄ backlog.md          # üìå CANONICAL ‚Äî all tasks live here
‚îú‚îÄ‚îÄ sprint-current.md   # üéØ DERIVED ‚Äî active tasks only
‚îî‚îÄ‚îÄ done.md             # ‚úÖ DERIVED ‚Äî completed tasks archive
```

The agent primarily interacts with `backlog.md` via the `/backlog` workflow. `sprint-current.md` and `done.md` are derived views that keep the backlog clean.
