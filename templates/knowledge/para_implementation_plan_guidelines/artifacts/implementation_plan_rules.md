# Implementation Plan Guidelines & Rules

This document outlines the strict boundary and operational guidelines distinguishing between **Project-level Implementation Plans** and **Platform-level (Antigravity IDE) temporary Implementation Plans** to prevent AI agent confusion during development.

---

## 1. Plan Categorization & Boundaries

### 1.1 Project Implementation Plan (Durable)
*   **Storage Path:** Always resides inside the specific project directory:
    `Projects/<project-name>/artifacts/plans/` (e.g., `Projects/pageel-crm/artifacts/plans/v0.10.3-plan.md`).
*   **Creation Trigger:** Explicitly generated by developers or AI agents utilizing the `/plan create` workflow.
*   **Characteristics:**
    *   Defines technical specifications, architecture designs, API contracts, and implementation phases of the actual feature or project.
    *   Directly linked with **plan sidecar skills** and the **project roadmap**.
    *   Task breakdowns are mapped and synced to the project's authority tasks files (`backlog.md`, `sprint-current.md`).
*   **Lifecycle:** Persistent and version-controlled. Committed directly to the project repository for team collaboration.

### 1.2 Platform Plan & Tasks (Ephemeral)
*   **Storage Path:** Resides temporarily inside the active conversation's brain directory:
    - `~/.gemini/antigravity-ide/brain/<conversation-id>/implementation_plan.md` (for the plan)
    - `~/.gemini/antigravity-ide/brain/<conversation-id>/task.md` (for the task checklist)
*   **Creation Trigger:** Automatically enforced/monitored by the Antigravity IDE platform during Planning and Execution Modes.
*   **Characteristics:**
    *   Serves as a synchronization and consent gate between the AI agent, the user, and the platform.
    *   Monitored natively by the IDE runtime to manage execution gates.
*   **Lifecycle:** Ephemeral. Must NOT be committed to the code repository. Automatically archived or deleted upon session closure.

---

## 2. Mandatory Agent Protocols

1.  **Strict Isolation:**
    *   When tracking features, checking off checklist items, or updating sprint progress, agents MUST target the project's authority files (`backlog.md`, `sprint-current.md`, or the plans inside `artifacts/plans/`).
    *   Agents MUST only edit the temporary `brain/implementation_plan.md` and `brain/task.md` files when prompted by the IDE to satisfy platform checkpoints. To prevent duplication and ensure consistency, both temporary files MUST contain an absolute Markdown link pointing directly to the durable Project-level Implementation Plan (e.g., `[Project Plan](file:///absolute/path/to/project/artifacts/plans/plan-name.md)`).
    *   The tasks (checklist items) presented inside `brain/task.md` MUST be defined solely by linking to the Project Plan (e.g., `Tasks are defined in [Project Plan](file:///absolute/path/to/project/artifacts/plans/plan-name.md)`). The agent MUST NOT copy or duplicate the detailed checklist tasks from the Project Plan into `brain/task.md`; all active tasks MUST reside exclusively in the Project Plan. When reading or updating task progress, the agent MUST follow this link to view and work on the actual tasks in the Project Plan instead of writing, modifying, or checking off tasks directly in the temporary file.
2.  **No Ephemeral Referencing (KR7 Compliance):**
    *   Agents MUST NOT add the temporary platform `brain/implementation_plan.md` or `brain/task.md` paths to the `references` array of any Knowledge Item (KI).
    *   Only Project Implementation Plans are allowed as durable references.
3.  **Platform Tracker Exemption for Macro Documents:**
    *   Macro-level documents such as Roadmaps, Strategy documents, Specs, and Brainstorms (typically stored under `artifacts/plans/roadmap.md`, `docs/strategy/`, `artifacts/para-decisions/`, `artifacts/specs/` or containing `roadmap`/`strategy`/`brainstorm`/`spec` in their names) represent high-level orchestrations, requirements, or designs rather than detailed implementation plans.
    *   To prevent token wastage and system conflicts, these macro documents are **fully exempted** from Platform Tracker synchronization. The agent MUST NOT create or sync temporary platform `implementation_plan.md` or `task.md` files in the brain folder when active_plan references these macro files. Workflows (`/open` and `/plan`) MUST automatically bypass sync gates for these documents.
