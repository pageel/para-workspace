# Knowledge System Architecture

> **Version**: 1.7.4 | **Last reviewed**: 2026-04-03

## Overview

PARA Workspace v1.7.0+ introduces the **Knowledge System** — persistent, cross-session memory for AI agents via Knowledge Items (KIs). KIs are curated, distilled knowledge that persists across conversations and projects.

KIs live **outside** the workspace, in the KI Store managed by the Antigravity platform (`~/.gemini/antigravity/knowledge/`). PARA Workspace governs KI operations (schema, rules, lifecycle workflows) without owning the storage layer.

## Architecture

```text
┌─────────────────────────────┐    ┌─────────────────────────────┐
│  KI Store                   │    │  PARA Workspace             │
│  ~/.gemini/antigravity/     │◀──│  Governance Layer:          │
│  knowledge/                 │    │  ├─ ki.schema.json          │
│  ├─ {slug}/                 │    │  ├─ heuristics.md (H10)     │
│  │  ├─ metadata.json        │    │  ├─ rules/knowledge.md      │
│  │  ├─ timestamps.json      │    │  └─ workflows/para-knowledge.md  │
│  │  └─ artifacts/           │    │                             │
│  └─ .archived/              │    │  Index Layer:               │
│     (retired KIs)           │    │  └─ .para/knowledge/        │
│                             │    │     ├─ index.md (12-col)    │
│  (platform-managed)         │    │     └─ reports/audit-*.md   │
└─────────────────────────────┘    └─────────────────────────────┘
```

## KI Types

### System KIs (`para_*`)

Managed by PARA Workspace. Source of truth: `repo/templates/knowledge/`.

- Slug prefix: `para_` (reserved, enforced by KR3)
- Owner: `para`
- Lifecycle: template → `./para install` → KI Store → `./para update` (auto-sync)
- Example: `para_workspace_architecture_standards`

### User KIs

Created by users/agents. No prefix restrictions (except `para_` which is reserved).

- Owner: `user`
- Lifecycle: `/para-knowledge [topic]` → KI Store
- Scope: `workspace`, `project`, or `ecosystem`
- Naming convention: `project_{name}` for project-specific, or descriptive topic slug
  - Example: project `my-app` → `project_my_app`
  - See [Slug Naming Convention](./ki-anatomy.md#slug-naming-convention-v173)

## Context Flow (v1.7.2)

```text
Session Start                    During Session                 Session End
─────────────                    ──────────────                 ───────────
Platform auto-injects      →     Agent uses injected data  →   /end suggests
KI summaries into context        for scope matching             KI create/update
                                 (no file I/O needed)

/open Step 2.7:                  /plan Phase E:                /end Step 4.7:
"From injected KI data,          pitfall → Risks               cross-ref session
match scope to project"          playbook → Phase refs          with KI titles
```

Before v1.7.2, workflows read `.para/knowledge/index.md` to discover KIs. This was redundant because the platform already injects KI summaries. v1.7.2 removes these file reads.

## Schema

Each KI contains a `metadata.json` with graph-ready fields:

| Field | Type | Description |
| :-- | :-- | :-- |
| `title` | string | Descriptive title |
| `summary` | string | ≤800 chars |
| `version` | string | KI content version |
| `scope` | enum | `workspace` / `project` / `ecosystem` |
| `domain` | string | Open taxonomy (e.g., `workspace`) |
| `purpose` | enum | `context` / `reference` / `pitfall` / `playbook` |
| `owner` | enum | `para` (system) / `user` |
| `para_version` | string | PARA version at creation/update |
| `tags` | string[] | Free-form tags |
| `references` | object[] | File paths, conversation IDs |
| `relates_to` | string[] | KI ↔ KI relationships (graph edge) |
| `code_refs` | string[] | KI ↔ Code relationships (graph edge) |
| `concepts` | string[] | KI ↔ Concept relationships (graph edge) |

Full schema: [`kernel/schema/ki.schema.json`](../../kernel/schema/ki.schema.json)

## Index Schema (v1.7.2)

`.para/knowledge/index.md` — user-facing reference, auto-generated by `/para-knowledge`:

```markdown
| # | Title | Slug | Owner | Scope | Domain | Purpose | PARA Ver | Arts | Refs | Modified | Health |
```

12 columns. Abbreviations: Owner = `para`/`user`, Scope = `ws`/`proj`/`eco`.

## Taxonomy

Two-axis classification: **domain** (open) × **purpose** (fixed enum).

```text
              context      reference    pitfall     playbook
            ──────────── ──────────── ─────────── ──────────
workspace   PARA arch    kernel specs update bugs audit

operations  infra setup  CI/CD ref    env traps   rollback
content     writing std  style guide  SEO traps   publish
strategy    org context  roadmap      scope creep planning

```

## Knowledge Graph Seed (v1.7.2)

System KI templates now include **file-level `code_refs`** (16 entries) that map which files are affected when modifying the knowledge system. This is the foundational data for future Knowledge Graph capabilities:

```text
v1.7.2 (now)           v1.8.0 (Department)     v2.x (Context Intelligence)
────────────           ─────────────────       ──────────────────────────
code_refs seed    →    Department KIs with  →  Auto-discover code_refs
(16 files manual)      domain-based routing    Semantic search on graph
relates_to empty       relates_to populated    Dependency traversal
```

## Governance

| Rule | Description |
| :-- | :-- |
| **KR1** Write | Only `/para-knowledge` workflow can create/modify KIs |
| **KR2** Approval | All KI operations require user confirmation |
| **KR3** Namespace | `para_*` prefix reserved for system KIs |
| **KR4** Summary | Summary ≤ 800 characters |
| **KR5** Recovery | All operations undoable (archive, not delete) |
| **KR6** Lifecycle | System KIs sync from repo templates via CLI |

Validation: **H10** (11 clauses) in `kernel/heuristics.md`.

## System KI Template Sync

```text
repo/templates/knowledge/     CLI install/update       KI Store
para_workspace_*/         ─────────────────────→  ~/.gemini/antigravity/
├─ metadata.json              compare para_version    knowledge/para_*/
└─ artifacts/                 merge strategy:         ├─ metadata.json
                              template wins metadata   ├─ timestamps.json
                              union merge references   └─ artifacts/
                              keep user artifacts
```

CLI `update.sh` auto-syncs system KIs using a dual-gate strategy (v1.7.3):

1. **Version gate**: compare `para_version` in template vs installed metadata
2. **Content hash gate**: compare file hash (catches same-version hotfixes)

If either gate detects a change, the template overrides the installed KI (idempotent — no-op when both match).
