# Knowledge System Architecture

> **Version**: 1.7.3 | **Last reviewed**: 2026-04-02

## Overview

PARA Workspace v1.7.0+ introduces the **Knowledge System** вҖ” persistent, cross-session memory for AI agents via Knowledge Items (KIs). KIs are curated, distilled knowledge that persists across conversations and projects.

KIs live **outside** the workspace, in the KI Store managed by the Antigravity platform (`~/.gemini/antigravity/knowledge/`). PARA Workspace governs KI operations (schema, rules, lifecycle workflows) without owning the storage layer.

## KI Types

### System KIs (`para_*`)

Managed by PARA Workspace. Source of truth: `repo/templates/knowledge/`.

- Slug prefix: `para_` (reserved, enforced by KR3)
- Owner: `para`
- Lifecycle: template вҶ’ `./para install` вҶ’ KI Store вҶ’ `./para update` (auto-sync)
- Example: `para_workspace_architecture_standards`

### User KIs

Created by users/agents. No prefix restrictions (except `para_` which is reserved).

- Owner: `user`
- Lifecycle: `/knowledge [topic]` вҶ’ KI Store
- Scope: `workspace`, `project`, or `ecosystem`
- Naming convention: `project_{name}` for project-specific, or descriptive topic slug
  - Example: project `my-app` вҶ’ `project_my_app`
  - See [Slug Naming Convention](./ki-anatomy.md#slug-naming-convention-v173)

## Schema

Each KI contains a `metadata.json` with graph-ready fields:

| Field           | Type       | Description                                  |
|:----------------|:-----------|:---------------------------------------------|
| `title`         | string     | Descriptive title                            |
| `summary`       | string     | вүӨ800 chars                                   |
| `version`       | string     | KI content version                           |
| `scope`         | enum       | `workspace` / `project` / `ecosystem`        |
| `domain`        | string     | Open taxonomy (e.g., `workspace`)            |
| `purpose`       | enum       | `context` / `reference` / `pitfall` / `playbook` |
| `owner`         | enum       | `para` (system) / `user`                     |
| `para_version`  | string     | PARA version at creation/update              |
| `tags`          | string[]   | Free-form tags                               |
| `references`    | object[]   | File paths, conversation IDs                 |
| `relates_to`    | string[]   | KI вҶ” KI relationships (graph edge)           |
| `code_refs`     | string[]   | KI вҶ” Code relationships (graph edge)         |
| `concepts`      | string[]   | KI вҶ” Concept relationships (graph edge)      |

Full schema: [`kernel/schema/ki.schema.json`](../../kernel/schema/ki.schema.json)

## Governance

| Rule             | Description                                       |
|:-----------------|:--------------------------------------------------|
| **KR1** Write    | Only `/knowledge` workflow can create/modify KIs  |
| **KR2** Approval | All KI operations require user confirmation       |
| **KR3** Namespace| `para_*` prefix reserved for system KIs           |
| **KR4** Summary  | Summary вүӨ 800 characters                          |
| **KR5** Recovery | All operations undoable (archive, not delete)     |
| **KR6** Lifecycle| System KIs sync from repo templates via CLI       |

Validation: **H10** (11 clauses) in `kernel/heuristics.md`.

## System KI Template Sync

```text
repo/templates/knowledge/     CLI install/update       KI Store
para_workspace_*/         в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’  ~/.gemini/antigravity/
в”ңв”Җ metadata.json              compare para_version    knowledge/para_*/
в””в”Җ artifacts/                 merge strategy:         в”ңв”Җ metadata.json
                              template wins metadata   в”ңв”Җ timestamps.json
                              union merge references   в””в”Җ artifacts/
                              keep user artifacts
```

CLI `update.sh` auto-syncs system KIs using a dual-gate strategy (v1.7.3):

1. **Version gate**: compare `para_version` in template vs installed metadata
2. **Content hash gate**: compare file hash (catches same-version hotfixes)

If either gate detects a change, the template overrides the installed KI (idempotent вҖ” no-op when both match).
