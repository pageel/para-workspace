# v1.6.5 — Update Flow Fix

> **Released:** 2026-03-30

Bug fix release addressing 3 logic issues in the update pipeline identified during v1.6.4 brainstorm review.

## Changes

### Fixed

- **BUG-22:** `update.sh` now detects version direction using `semver_gte`. Displays ⏬ "Downgrade detected" instead of ⏫ "Upgraded" when workspace kernel is ahead of repo VERSION. Migration is skipped in downgrade scenarios.
- **BUG-23:** `migrate.sh` now tracks `MIGRATION_RAN` flag. History log (`history.log`) is only written when migration steps actually execute, not when all version gates are skipped.
- **BUG-24:** `update.md` workflow decision gates now include "Kernel > Repo VERSION" row with warning about pre-pull comparison. Clarified that repo VERSION is read before `git pull` and may be stale.

### Changed

- `update.sh` now loads `validator.sh` library at startup (for `semver_gte` access).
- `install.sh` added documentation comment explaining bash dynamic scoping for `count`/`updated` variables in `sync_directory_recursive()`.

## Files Modified

| File | Change |
| :--- | :----- |
| `cli/commands/update.sh` | Load validator.sh, version direction detection |
| `cli/commands/migrate.sh` | MIGRATION_RAN tracker, conditional history.log |
| `cli/commands/install.sh` | Documentation comment (dynamic scope) |
| `templates/common/agent/workflows/update.md` | Decision gates + pre-pull note |

## Migration

No migration steps needed. This is a PATCH release — `para update` will sync automatically.
